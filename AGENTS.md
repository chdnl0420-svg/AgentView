CRITICAL:TDD기반 작업을 진행한다.
- TDD(Test Driven Development)란, 테스트 주도 개발을 의미한다. 이는 소프트웨어 개발 방법론 중 하나로, 먼저 테스트 케이스를 작성하고 그 테스트를 통과하기 위한 최소한의 코드를 작성하는 방식으로 개발을 진행하는 방법이다.
- TDD의 주요 단계는 다음과 같다:
  1. 테스트 작성: 먼저, 개발자는 새로운 기능이나 개선하려는 기능에 대한 테스트 케이스를 작성한다. 이 테스트는 아직 구현되지 않은 기능을 검증하기 위한 것이다.
	- 테스트는 스크린샷과 화면 클릭을 포함하여 작성되어야 한다. 이는 테스트가 실제로 사용자 인터페이스와 상호작용하는 것을 검증하기 위함이다.
  2. 코드 작성: 테스트 케이스가 실패하는 것을 확인한 후, 개발자는 테스트를 통과하기 위한 최소한의 코드를 작성한다. 이 단계에서는 기능이 완전히 구현되지 않아도 된다.
  3. 리팩토링: 테스트가 통과하면, 개발자는 코드를 리팩토링하여 더 깔끔하고 효율적으로 만든다. 이 단계에서는 코드의 구조를 개선하지만, 기능은 변경하지 않는다.
  4. 반복: 이 과정을 반복하여 새로운 기능을 추가하거나 기존 기능을 개선한다.
CRITICAL:구현 단계에서는 유지보수가 용이하도록 코드를 작성한다.
	1. 코드는 명확하고 간결하게 작성되어야 한다. 불필요한 복잡성을 피하고, 가독성을 높이는 것이 중요하다.
	2. A를 수정하는데 B가 영향을 받지 않도록 모듈화된 구조로 코드를 작성한다. 이는 코드의 재사용성을 높이고, 유지보수를 쉽게 한다.
	3. 단일 파일에 모든 코드를 작성하는 것을 피하고, 기능별로 파일을 분리하여 관리한다. 이는 코드의 가독성을 높이고, 협업 시 충돌을 줄이는 데 도움이 된다.
	4. 작업진행중 구조가 복잡해질 경우, 리팩토링을 통해 코드를 개선한다. 리팩토링은 코드의 기능을 변경하지 않으면서 구조를 개선하는 과정이다. 이를 통해 코드의 유지보수성을 높이고, 버그를 줄일 수 있다.
CRITICAL:모든 작업내용을 문서화한다.
	1. 보고서 및 계획 문서는 HTML로 작성한다.
	2. 그외 모든 산출물을 md파일로 작성한다.
	3. 새로운 작업을 시작할때는 작업의 목적과 계획을 명확히 적는다.
	4. 작업이 중간에 중단되어도 작성된 산출물을 통해 작업을 이어갈 수 있도록, 문서에는 작업의 목적, 진행 상황, 다음 단계에 대한 명확한 설명이 포함되어야 한다.
